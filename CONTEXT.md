# DataLens Backend — Domain Language

## Documents
A text file uploaded by a user (PDF, DOCX, HTML, TXT). Goes through ingestion (parse → OCR → PII detect → chunk → index) before becoming searchable. Has versioning, ACLs, and status tracking (`ready`, `processing`, `error`). Stored as `Document`.

## Ingestion Pipeline
Transforms a raw Document into queryable chunks. Steps: parse raw bytes into text → OCR fallback for scanned PDFs → PII detection and redaction → chunking (recursive/fixed/semantic) → vector indexing + BM25 indexing → status update. Runs via Celery worker asynchronously.

## Chunk
A slice of a Document's text, produced by the Chunker during ingestion. Has boundaries, a parent Document reference, and text content. The unit of retrieval — the retriever returns Chunks, not whole Documents.

## Retrieval
Finding Chunks relevant to a user Query. Uses hybrid retrieval: dense (FAISS vector similarity) + sparse (BM25 keyword) fused via Reciprocal Rank Fusion. Optionally preceded by Query Expansion or HyDE. Optionally followed by a Reranker.

## Query Expansion
An LLM decomposes a user query into 2–3 sub-queries to improve recall. A pre-retrieval step.

## HyDE (Hypothetical Document Embeddings)
An LLM generates a hypothetical answer to the user query, then embeds that hypothetical text for similarity search. A pre-retrieval step that improves dense retrieval when the query vocabulary differs from document vocabulary.

## Reranker
Re-scores retrieved Chunks for relevance. Either a local cross-encoder (`sentence-transformers`) or the Cohere Rerank API. A post-retrieval step.

## Context Assembly
Builds the final prompt context window from retrieved and reranked Chunks, respecting the LLM's token budget. Truncates Chunks and adds metadata (source document, anchor).

## Chat
The full end-to-end pipeline: Query → optional safety check → semantic cache lookup → retrieval → reranking → context assembly → LLM invocation → cost tracking → cache write → optional follow-up generation → response. Can run in streaming (SSE) or non-streaming (JSON) mode.

## Conversation
A sequence of Messages between a User and the assistant, grouped under a single conversation ID. Has a title (auto-generated by LLM), timestamps, and optional sharing.

## Message
A single turn in a Conversation: role (`user` or `assistant`), text content, and metadata (sources, token counts). User Messages can include file references.

## Semantic Cache
A Redis-backed cache that stores (query embedding → answer) pairs. On a cache hit, the pipeline skips retrieval and LLM invocation, returning the cached answer directly. Reduces latency and cost for repeated or similar queries.

## Safety Layer
A configurable pipeline of input/output safety checks: input guardrails (SQL injection, prompt hijacking patterns), prompt injection detection (LLM-based), grounding verification (is the answer grounded in retrieved context?), and citation validation. Retryable with relaxed thresholds.

## Share Token
A one-time-use token that grants access to a specific Conversation. Supports optional password protection and view counting. Used for sharing chat results externally.

## Connector
An adapter that pulls documents from an external data source (local filesystem, S3, Google Drive, Notion). Each Connector has configuration credentials stored as `ConnectorConfig`.

## Model Router
Decides which LLM (provider + model) serves a given Query based on rules: chat history length, safety score, query complexity. Currently rule-based.

## Evaluation
RAGAS-based quality evaluation using a Golden Dataset (ground-truth Q&A pairs). Runs as a Celery task triggered by an Experiment. Produces per-question `ExperimentResult` rows and aggregate metrics (faithfulness, answer relevance, context relevance).

## Golden Dataset
A curated set of question-answer pairs with optional reference documents. Used as ground truth for evaluation experiments.

## Retention Policy
A data lifecycle policy that defines how long Conversations, Messages, Feedback, and Experiments are kept. A daily Celery beat task enforces the policy by deleting expired data.

## Audit Log
An immutable trail of user actions (login, chat, document operations, sharing, GDPR actions). Stored as `AuditLog`. Queryable by admins with filtering and CSV export.

## RAG Settings
Per-user feature flags controlling which RAG pipeline components are active (HyDE, query expansion, reranker, semantic cache, safety layer, model routing). Can be set per user via `RAGSettings` and also have global defaults via env vars.

## Provider
An external LLM API (Groq, MiniMax, OpenAI). Each Provider has a model name, API key, base URL, and circuit breaker for resilience.

## Vector Store
The dense embedding index. Supports FAISS (default), Chroma, and Milvus. Used by the dense retriever and the semantic cache for similarity search. A module-level singleton, lazy-initialized.
